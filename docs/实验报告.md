# OPDR1 实验报告：小模型 Agentic Search 上 RL 与在策略蒸馏的对比

> **一句话** ：在 agentic search 任务上，用同一个 3B 学生模型，系统对比 **GRPO（强化学习）** 与 **OPD（在策略蒸馏）** 两条后训练路线的效果与能力上限。

> 学生 = Qwen2.5-3B-Instruct；老师 = ZeroSearch_google_V2_Qwen2.5_7B_Instruct；搜索后端 = ZeroSearch 模拟检索（`search_mode=simulate_sft`，搜索结果由模拟器 LLM `Simulation_LLM_google_3B` 生成，prompt 硬编码为 "You are the Google search engine"，**不调用任何真实搜索 API**；`search_engine` 参数在 verl 管线中不被消费，见 §2.1 注）；训练框架 = veRL + vLLM/sglang。
>

## 1. 研究问题与两条路线

同一个小模型（Qwen2.5-3B-Instruct），要它学会"推理—检索交错"的多轮 agentic search 行为，两条后训练路线各能走到哪：

* **GRPO（RL 路线）** ：只给 outcome reward（F1 规则奖励），用 group-normalized advantage 做策略梯度。不依赖老师，理论上能逼近甚至局部超过任何单一参考模型——上界由任务本身与学生容量决定。
* **OPD（在策略蒸馏路线）** ：学生 on-policy 采样自己的轨迹，7B 老师只做前向、给逐 token log-prob，最小化反向 KL `KL(π_student ‖ π_teacher)`。不需要 reward、不需要 critic/advantage——工程更简单，但 **能力上界被老师封死** （学生在拟合老师分布，系统性超过信号源在信息论上不成立）。

在 veRL 里，OPD 与 GRPO 的差别几乎就是"损失函数改一处"：把参考模型从"学生的冻结副本"换成 7B 老师，把 KL 从小权重正则升为主损失，把 advantage 项权重置 0。这也是本实验能以最小改动、在同一套代码里公平对比两条路线的前提。

## 3. OPD 实现与训练过程

### 3.1 最小改动实现 OPD

在 veRL GRPO 基础上，用 `loss_mode: grpo | opd` 开关切换， **不破坏原 GRPO 路径** ：

| 文件                                     | 改动                                                 | 说明                                                |
| ---------------------------------------- | ---------------------------------------------------- | --------------------------------------------------- |
| `verl/trainer/config/ppo_trainer.yaml` | 新增 `loss_mode: grpo`                             | 默认 grpo，向后兼容                                 |
| `verl/trainer/config/ppo_trainer.yaml` | 新增 `ref.model_path: null`                        | null 时回退到 actor path（即 GRPO 行为）            |
| `verl/workers/fsdp_workers.py`         | ref 模型路径读 `config.ref.model_path`，`or`回退 | 不破坏 GRPO                                         |
| `verl/workers/actor/dp_actor.py`       | `select_keys`在 OPD 模式下也取 `ref_log_prob`    | `getattr`安全访问                                 |
| `verl/workers/actor/dp_actor.py`       | 新增 `if loss_mode == 'opd':`分支                  | KL 对老师为主损失；pg_loss / clipfrac / ppo_kl 置零 |
| `train_opd.sh`                         | 从 `train_grpo.sh`派生的新脚本                     | 新增 `TEACHER_PATH`；`kl_loss_coef=1.0`         |

 **未改动** （确认无需改）：`ray_trainer.py`（`compute_ref_log_prob` 始终被调用）、`core_algos.py`（`kl_penalty()` 直接复用）、`megatron_actor.py`（本实验走 FSDP）。

### 3.2 三个正确性约束（动手前已确认）

1. **tokenizer 一致** ：老师与学生同为 Qwen2.5 系，逐 token 反向 KL 对齐成立。
2. **masking 只作用于学生生成的 token** ：`state_masking=True` 把 `<information>...</information>`（检索/工具返回内容）排除在蒸馏之外，避免学生去拟合模拟器吐出的文本。
3. **KL 估计器** ：复用 veRL 已有的 `low_var_kl`（Schulman k3 低方差估计器），对应最小化 `KL(π_student ‖ π_teacher)` 的反向 KL 方向。

### 3.4 关于 GRPO 路线：为何采用官方模型

GRPO 路线在本机完成了  **smoke 冒烟测试** ，确认了训练通路可跑通（模拟器拉起、rollout、reward、advantage、actor 更新全链路无崩溃）。但由于本轮自训的数据量与步数都很小、不足以产出一个有代表性的 GRPO 模型， **最终对比直接采用 ZeroSearch 官方发布的训好 3B 模型** （`ZeroSearch_google_V2_Qwen2.5_3B_Instruct`，官方训练 203 步、14B 模拟器）。

由此带来一处 **必须明确的不对称** ：官方 GRPO 模型与本实验 OPD 模型在"训练主体、模拟器规模（14B vs 3B）、训练步数（203 vs 400）"上均不同。

**因此本实验能回答与不能回答的问题要分清：**

* ✅  **能回答** ：GRPO（RL）路线训出来的 3B 能达到什么能力上限、与 7B 老师的差距有多大、在单跳/多跳上的分布如何——因为这对比的是两个已训好的模型，结论不依赖本实验的训练过程。
* ❌  **不能回答** ：GRPO 与 OPD 谁"性价比更高 / 更省成本"——两臂训练条件不一致，且本实验未自训 GRPO、没有可比的训练成本数据。**本报告不下"性价比"结论。**

## 5. 结果分析

### 5.1 整体排序符合理论：基线 < OPD < GRPO ≤ 老师

四个模型的高低次序与两条路线的机制预期完全一致，是本实验最稳的一条结论：

* **OPD 以老师为上界** ：反向 KL 蒸馏在信息论上不可能系统性超过信号源，OPD 均值（EM 0.327）低于老师（0.457）符合预期。
* **GRPO 不受老师约束** ：outcome reward 直接优化任务指标，可以逼近老师并在局部超过它。

### 5.2 GRPO 3B 追平 7B 老师，并在多跳反超（核心结论，且可靠）

GRPO 3B 与 7B 老师 EM 均值仅差 3%（0.444 vs 0.457），F1 差 5%。分任务看：

* **多跳反超老师** ：2Wiki（0.412 vs 0.378，+9%）、MuSiQue（0.294 vs 0.278，+6%）。
* **单跳老师仍领先** ：NQ / TriviaQA / PopQA 老师 +1~3%。
* **Bamboogle 老师领先最多** （0.342 vs 0.264，+30%）——它是最难的组合多跳，样本少，对参数化知识与长链路都吃紧，7B 优势最大。

 **机制解释** ：outcome reward 奖励的是"最终答对"，它塑造的是**推理—检索链路**而非记忆事实。多跳任务的瓶颈在链路（要不要再搜一轮、如何组合线索），正是 RL 能直接优化的；单跳任务瓶颈在事实覆盖，7B 天然更强。所以"多跳反超、单跳落后"是理论上可预期的漂亮结果——RL 学到的是推理能力而非死记。

> **可靠性说明** ：此结论对比的是两个官方训好模型（官方 GRPO 3B vs 官方 7B 老师），本实验只做了统一口径评测，结论不依赖本实验的训练过程，因此是本工作里最硬、最可对外陈述的部分。

### 5.3 OPD 达到老师约 72%：原因辨析（含一处归因修正）

OPD 3B EM 0.327，为老师（0.457）的约 72%，仍差 13 个 EM 点。原因需要分清主次：

**① lr 与 warmup 过保守（成立，主因之一）。** `lr=1e-6` + `lr_warmup_steps_ratio=0.95`，意味着 400 步里前 ~380 步都在线性爬坡，平均有效 lr 极低，学生远未走够。这是纯超参问题，最容易验证也最容易改。

**② 蒸馏信号质量 / 模拟器规模不一致（待验证，权重低于 ①）。** on-policy 蒸馏的前提是"老师在学生自己生成的轨迹上能给出可信的 log-prob"。本设定中学生 OPD rollout 用 `Simulation_LLM_google_3B`（3B 模拟器）生成检索内容，而老师 `ZeroSearch_google_V2_Qwen2.5_7B_Instruct` 训练时用的模拟器规模未知（ZeroSearch 官方内部配置，§2.1 表中官方 GRPO 训练时用的是 14B 模拟器，老师很可能也是 14B 量级）。若老师训练时的模拟器输出质量显著高于学生的 3B 模拟器，老师对学生 3B 模拟器轨迹打的 log-prob 可能噪声偏大，反向 KL 梯度方向不够干净。但此条缺乏直接证据（不知老师训练时的模拟器确切规模），归因权重应低于 ①。

> **关于早先文档里的"google 模拟器 + google 老师 + wiki 语料三方不匹配"归因——经代码追踪后弃用。** `search_engine` 参数在 verl 训练/评测管线里是死代码（详见 §2.1 注），`simulate_sft` 模式下搜索结果由模拟器 LLM 生成、prompt 硬编码为 "You are the Google search engine"，与 `search_engine` 取值无关。因此不存在 google/wiki 语料维度的分布偏移。真实的潜在偏移只可能来自模拟器**模型规模**（3B vs 14B）差异，即 ② 所述。

**③ 关于"OPD 无差别蒸馏所有 rollout、信号效率低于 GRPO"这一说法——建议弃用。** 早先文档曾把"OPD 在全部 5 条 rollout 上蒸馏、GRPO 只奖励高优势样本"当作 OPD 分数低的原因之一。这个归因在理论上站不住：

* OPD 的目标本就是 **分布匹配** ——在学生访问到的整个状态分布上把策略拉向老师，"覆盖所有轨迹"是 on-policy 蒸馏相对 SFT 的 **设计优势** ，不是信号浪费。
* GRPO 只在高优势样本上给强信号，是因为它没有老师、只能靠 reward 的相对排序找方向，这是它的**约束**而非效率优势。

因此，OPD 分数低的真实原因是 **训练预算（lr/步数，①）+ 可能的模拟器规模偏移（②，待验证）** ，与"信号覆盖太广"或"google/wiki 语料不匹配"无关。

### 5.4 OPD 全面超过基线

OPD 蒸馏方向正确、有效：7 个 benchmark 全部提升，EM 均值 0.208 → 0.327（+57%）。基线 3B-Instruct 几乎不具备 agentic 搜索能力，反向 KL 把它明显推向了"会检索、会用检索结果"的方向。多跳上的相对增益尤其大（MuSiQue +149%、Bamboogle +87%）——但需注意这是 **相对百分比的放大效应** （基线多跳分数接近地板），不代表 OPD 在多跳上学得特别充分。

### 5.5 稳健性与统计噪声提醒

* 3B 系列每 benchmark 仅 48 条，Bamboogle 等小集上几个点的差异可能落在评测随机性内，单点对比需谨慎。
* 本轮为单次运行、未做多 seed，因此细粒度分任务的名次差异不宜过度解读；稳的是**均值层面的整体排序**与 **多跳/单跳的定性规律** 。

| 调整                                       | 具体动作                                                                            | 目的                                               |
| ------------------------------------------ | ----------------------------------------------------------------------------------- | -------------------------------------------------- |
| **A. 降低 warmup / 提高 lr**         | `lr_warmup_steps_ratio`0.95 → 0.1~0.2，或 `lr`1e-6 → 2e-6~5e-6               | 让学生在有限步数内真正走够，剥离"没走够"这一混淆项 |
| **B. 验证模拟器规模一致性**          | 查证老师训练时用的模拟器规模；若与学生的 3B 模拟器差距大，考虑换 14B 模拟器重训 OPD（注：`search_engine` 参数在 verl 管线中是死代码，调它无意义，见 §2.1 注） | 消除模拟器规模偏移，让老师对学生轨迹的 log-prob 可信 |
| **C. 增加步数**                      | 400 步 → 800~1000 步                                                               | 观察 OPD 是否随步数继续爬升                        |
| **D.（可选）自训 GRPO 做同条件对照** | 用与 OPD 相同的模拟器/步数自训一个 GRPO 3B                                          | 唯一能让"路线对比 / 成本对比"成立的前提            |

### 6.2 预期结果

* **调整 A（lr/warmup）** ：若 72% → 明显上升（例如 80%+），说明本轮主要瓶颈是训练预算不足；若几乎不动，说明瓶颈在信号质量（模拟器规模偏移），据此可锁定主因。**无论涨或不涨，都能让归因从"两个并列猜测"收敛到确定结论。**
* **调整 B（验证/统一模拟器规模）** ：若查证发现老师训练用 14B 模拟器、学生用 3B，换成 14B 重训后预期 OPD 能进一步接近老师，反向 KL 曲线更平稳、下降更充分；若老师本就用 3B 量级模拟器训的，则本项无操作空间，瓶颈完全落在 A/C。
* **调整 C（加步数）** ：预期 OPD 分数随步数单调上升并逐步逼近饱和，但 **饱和点不超过老师** ——这是反向 KL 的信息论上界，属于要呈现的结论而非缺陷。
* **调整 D（同条件自训 GRPO）** ：预期仍是 GRPO ≥ OPD（上限更高），但只有拿到同条件的两臂训练开销后，才能谈"哪条路线在给定预算下更划算"。

### 6.3 预期结论

* **能力上限** ：RL（GRPO）> OPD。RL 不受老师约束、可突破蒸馏天花板并在多跳反超大模型；OPD 封顶于老师能力。此结论方向明确，本轮已验证，后续调整只会强化。
* **OPD 的定位** ：一条**工程更简单、无需 reward 设计、收敛更平稳**的路线，适合"已有一个强老师、且能保证老师-学生分布一致"的场景；其效果上限取决于老师质量与分布匹配程度，而非蒸馏本身的信号覆盖方式。
* **暂不下的结论** ："OPD 比 RL 更省成本 / 性价比更高"。需要调整 D（同条件自训 GRPO + 统一成本口径）之后才能回答。在此之前，对外只陈述"能力上限"与"工程复杂度"两个维度的对比。

## 8. 附：术语与口径说明

* **OPD（On-Policy Distillation，在策略蒸馏）** ：学生自采样轨迹、老师只给 log-prob、最小化反向 KL。区别于 SFT 蒸馏（在老师轨迹上模仿，存在 distribution shift）。
* **反向 KL `KL(π_student ‖ π_teacher)`** ：本实验用 `low_var_kl`（Schulman k3）估计，方向为把学生拉向老师；其能力上界即老师。
* **outcome reward** ：只对最终答案的正确性（F1/EM）给奖励，不对中间步骤打分。
* **state_masking** ：检索/工具返回内容（`<information>...</information>`）不计入损失，只在学生自己生成的 token 上算 loss。
* **`search_mode` vs `search_engine`** ：`search_mode` 决定搜索行为（`simulate_sft` = 模拟器 LLM 生成；`google` = 真 SerpAPI；`wiki` = 真 wiki 检索服务），是实际生效的参数。`search_engine` 在 verl 训练/评测管线里**不被任何 Python 代码消费**（唯一消费点是独立的 `inference.py`），在 `simulate_sft` 模式下设置它无行为效果——本报告所有"google/wiki 模拟器"的表述均指**模拟器模型的训练分布**（如 `Simulation_LLM_google_3B` 在 google 风格搜索结果上训练），不是 `search_engine` 参数取值。
* **成本口径** ：本报告不给出 RL vs OPD 的成本对比结论（GRPO 用官方模型、未自训、条件不对称）。训练/评测耗时见 §3.5，仅作记录、不作路线成本对比依据。
* **样本量** ：3B 系列 48 条/benchmark，7B 老师 195 条全量；单次运行、未做多 seed。分任务细粒度差异请结合此口径谨慎解读。
